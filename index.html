Rails REST API that gives clients costume contest results in json

What is this?

REST API
Ruby on Rails

This limited REST API example-project gives clients costume contest results in json format. The backend is Ruby on Rails with help from the active_model_serializers gem and the rack-cors gem. To see a sample AngularJS web app consume this, click here

Endpoints?

Three endpoints:

  1. get a list of contests
  2. get a single contest, with all of that contest’s categories, and with the entries for those categories
  3. get a single category from a contest, with the entries for that category

Example URLs (as viewed in a web browser, ‘real’ clients would leave off the .json extension and ask for json in a request header):

  • http://rest-api-costume-contest.herokuapp.com/api/v1/contests.json
  • http://rest-api-costume-contest.herokuapp.com/api/v1/contests/1.json
  • http://rest-api-costume-contest.herokuapp.com/api/v1/contests/1/categories/1.json

Why did I make this?

This mini-project was an effort to learn to REST/API, and how that could be implemented and served from a Ruby on Rails backend.

What technology/skills were learned/retrieved?

  • REST and APIs (concepts, Rails implementation)
  • Rails, associations (has many through), rails routing, namespacing, scaffolding, and the active_model_serializers gem
  • json
  • CORS

Why make an API about costume contest results?

I wanted to do a costume contest app in a full blown Rails app, and this api project was a way to try the idea. After this, do I still want to make a full app? No - there is a limited audience, it’s more complex than I thought, and contest creators seem happy using blogs and tumblr to record and publish their contests.

Limitations:

  • No POST, DELETE or PUT actions. In my self-imposed time-constraint, implementing just the GET requests was challenging. In this rails app, the API and backend rails models are not 1 to 1. A 1 to 1 API may be appropriate in some apps, but in this case a meaningful API response for costume contest results involves assembling associated data from a variety of models.
  • Another limitation is that this app does NOT have a meaningful front-end - this project was meant for serving a REST API.

What more could be explored?

  • Different REST verbs (this only does GET for now)
  • Security and Authentication
  • Compliance to jsonapi.org standards
  • Error Handling
  • Documentation
  • Regression Tests
  • Versioning - although this API has namespace versioning, a second version has not been implemented yet
